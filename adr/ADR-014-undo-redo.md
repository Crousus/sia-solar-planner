# ADR-014: App-Wide Undo / Redo via Custom Zustand Middleware

- **Status:** Accepted
- **Date:** 2026-04-21
- **Requirement:** Give users a reliable Cmd/Ctrl+Z across every project-data mutation — roofs, panels, strings, inverters, drags, renames — without corrupting the captured satellite image, without bloating memory, and without losing history across JSON save/load.

## Context
The app had grown past the point where a single misclick was recoverable. Drawing a roof, dragging a group of panels, re-assigning a string, and then realizing the last action was wrong meant manual reconstruction. Third-party zustand undo libraries exist but each failed one of our hard constraints:

1. **Captured satellite image must stay out of history.** After Lock Map (ADR-007), `mapState.capturedImage` holds a multi-MB base64 PNG. Snapshotting it per mutation would blow past the localStorage quota within a handful of edits and crash tabs.
2. **Rapid drags must coalesce.** A single panel drag produces dozens of `updatePanelPosition` calls at 60fps. Each one as a separate history entry would mean "undo" moves the panel back by one frame.
3. **UI state must bypass the record path.** Tool-mode changes, selection changes, active-string flips — these aren't "edits" from the user's mental model and pushing them onto the stack would bury the real undo targets.
4. **History must round-trip through JSON save/load.** Users export projects mid-session; re-importing and losing their stack of reversible edits would be a regression.
5. **Refreshing the tab must start clean.** localStorage persistence shouldn't carry history across reloads — the stored `past` could reference UI state that doesn't match the reloaded DOM, and the user's mental model of "refresh = fresh session" is stronger than "refresh = resumable undo."

Off-the-shelf middleware treats the whole store state uniformly; our requirements demand per-field policy.

## Decision

1. **Custom middleware at `src/store/undoMiddleware.ts`.** `undoable(...)` wraps the store's `set` and classifies every call through an explicit `ACTION_POLICY: Record<ActionName, 'bypass' | 'record' | 'clear-history' | 'load-history'>` map. Every `set(partial, replace, actionName)` call in `projectStore.ts` passes an action name as the third arg; unknown action names throw in dev so a newly-added action can't silently default to the wrong policy. Four policies, not a boolean — because `resetProject` and `loadProject` need *different* treatments than either "record" or "bypass" (they clear or replace the stacks entirely, rather than pushing onto them).

2. **`UndoableSlice` is a narrow projection, not the whole store.** `buildSlice(project)` extracts only `{ name, panelType, roofs, panels, strings, inverters }` — everything under `project` *except* `mapState`. `mapState` is reassembled on restore from the store's current value, which means the captured image never enters a snapshot and undoing a roof edit never reverts the satellite backdrop. `UIState` fields (selection, tool mode, drag ephemera, `splitCandidateRoofId`) are likewise outside the slice and bypassed.

3. **500ms time-coalescing with a two-field handshake.** Coalesce-eligible actions (`updatePanelPosition`, `renameRoof`, `setRoofTilt`, `assignPanelsToString`, etc.) cooperate via two store fields: `_pendingCoalesce` (the write target — what the *next* set will try to coalesce against) and `lastActionSig` (the comparison target — the signature that arrived with the last recorded step). If the incoming action's signature matches `lastActionSig` and under 500ms have elapsed, the snapshot is suppressed rather than pushed. The signature isn't just the action name — dragging two different panels within 500ms must *not* collapse into one step, so each coalescing action chooses its own key (e.g., `renameRoof` keys by roof id; `updatePanelPosition` keys by panel id). The middleware exposes `setCoalesceKey(set, actionName, key)` for this. Settled drags (`moveGroup`) intentionally *don't* coalesce — they fire once at the end of a pointer gesture, so each is already atomic.

4. **Past/present/future triplet, 100-entry cap.** On a `record` set, the middleware shifts the current slice into `past` (`past.slice(-99).concat(snapshot)`) and clears `future`. Undo moves the top of `past` into `future` and applies `applyUndo` as a pure `(S) => Partial<S>`; redo is symmetric. The 100-step cap keeps localStorage bounded; older entries fall off the bottom — an intentional trade, since 100 actions back is already past the point of useful undo for spatial work.

5. **`canUndo` / `canRedo` as mirror booleans, not selectors.** Derived-on-read selectors would rerender toolbar buttons on *every* store change. Materializing them as fields flipped in lockstep with `past`/`future` means the Toolbar subscribes to scalar booleans and only rerenders when the button enabled-state actually changes.

6. **Post-restore UI-ref cleanup via `cleanUiRefs`.** Undoing a `deleteRoof` brings the roof back, but the selection state it referenced may now dangle (`selectedRoofId` pointing at an id that no longer exists post-restore, or persists across an ID-reassignment). `cleanUiRefs(uiState, restoredSlice)` nulls any selection/active-string/active-group ref whose target isn't in the restored slice. The same helper is reused by cascading deletes (`deleteRoof`, `deleteString`, `deleteInverter`) — a post-delete slice runs through `cleanUiRefs` to catch dangling refs the explicit per-delete-branch clears had missed (e.g., `splitCandidateRoofId` across a roof deletion, `activePanelGroupId` when the group's panels all lived on the deleted roof).

7. **Dev-only referential-integrity assertion.** After every restore, if `import.meta.env.DEV`, `assertReferentialIntegrity(slice)` walks the slice and flags any `panel.roofId` with no matching roof, any `panel.stringId` with no matching string, any `string.inverterId` with no matching inverter. This is a safety net for middleware bugs, not a user-facing check — in prod, the cost-per-restore goes to zero. Vite's `import.meta.env.DEV` rather than `process.env.NODE_ENV` because the latter isn't injected in Vite browser builds.

8. **Persist wraps undoable, not the other way around.** `persist(undoable(creator))` means persist's `partialize` sees the full store including history — but we explicitly `partialize` to strip `past`, `future`, `lastActionSig`, `_pendingCoalesce`, `canUndo`, `canRedo`. This is the "refresh = clean history" constraint. If we inverted the ordering, persist would restore history fields first and then undoable would never get a chance to reset them.

9. **JSON format bumped to v2; v1 still loadable.** Export now writes `{ version: 2, project, history: { past, future } }`; import dispatches: if `parsed.version === 2`, call `loadProject(parsed.project, parsed.history)`; otherwise treat the whole object as a v1 raw `Project` and call `loadProject(parsed)` which leaves history empty. The `loadProject` signature grew a second optional arg for this, and the `load-history` policy in the middleware is what splices the restored stacks back into place atomically.

## Consequences

- **Pros:** Undo works uniformly across every project-data action (≈25 actions) without per-action plumbing; the middleware is the only place that needs to know about history. Captured satellite images never enter history, so memory stays bounded. Rapid drags and renames collapse to single steps, matching user intuition. Exported JSON files carry history across sessions (if the user wants it), while tab refreshes give a clean slate. The `ACTION_POLICY` map acts as a typed registry that forces any new action to explicitly declare its history semantics at definition time — making "I forgot to handle undo for this feature" a compile-time question, not a runtime one.

- **Cons:** Every `set()` call site in `projectStore.ts` now requires a third arg (action name) — a one-time port cost (~33 sites) and an ongoing discipline. The coalescing two-field handshake (`_pendingCoalesce` + `lastActionSig`) is subtle; getting the key wrong for a new coalescing action will either over-coalesce (two distinct edits merged) or under-coalesce (every frame of a drag recorded). The 100-entry cap is silent — there's no UI to tell the user they've hit it, and the oldest entry just evaporates. History round-tripping through JSON bloats export size linearly with stack depth, and nothing deduplicates common prefixes between snapshots (each `UndoableSlice` is a full copy of the undoable projection, structurally shared within a run but not across serialization).
